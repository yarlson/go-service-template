# go-service-template

A small production baseline for HTTP services written in Go. It includes one
complete users feature to demonstrate contract validation, authentication,
business logic, PostgreSQL persistence, migrations, tests, and graceful runtime
behavior without prescribing a web framework. Its asynchronous contract follows
the SNS/SQS envelope used by Lokalise Node services, while service-private jobs
use the existing PostgreSQL database.

## Stack

- Go 1.26.5 and the standard-library `net/http` router
- OpenAPI 3.0.3 with strict server types generated by `oapi-codegen`
- AsyncAPI 3.0 with a Lokalise-compatible SNS/SQS message envelope
- PostgreSQL through `pgx/v5`, with SQL queries generated by `sqlc`
- Transactional background jobs and schedules through River
- AWS SDK v2 publication to SNS for cross-service events
- UUIDv7 for persisted identifiers and server-generated request IDs
- Embedded SQL migrations through `golang-migrate`
- OIDC JWT bearer authentication through `go-oidc`
- Structured logging through `log/slog`
- Optional OTLP tracing and Prometheus metrics through OpenTelemetry
- Testify assertions and isolated PostgreSQL integration tests through Testcontainers
- golangci-lint v2, `govulncheck`, race tests, Docker, and GitHub Actions

All Go developer tools are versioned in `go.mod` and run through `go tool`.
Nothing needs to be installed globally.

## Start locally

Requirements: Go 1.26.5, Docker with Compose, and Make.

```sh
make bootstrap
make dev
make worker
```

`bootstrap` creates `.env`, starts PostgreSQL, and applies migrations. Local
configuration disables authentication; production configuration cannot do so.
The test suite verifies that `.env.example` and the typed configuration remain
in sync.

Create and fetch a user:

```sh
curl -i -X POST http://localhost:8080/v1/users \
  -H 'Content-Type: application/json' \
  -d '{"email":"person@example.com"}'

curl -i http://localhost:8080/v1/users/<user-id>
```

Submit and inspect a durable user import:

```sh
curl -i -X POST http://localhost:8080/v1/user-imports \
  -H 'Content-Type: application/json' \
  -d '{"emails":["one@example.com","two@example.com"]}'

curl -i http://localhost:8080/v1/user-imports/<import-id>
```

The API stores the import and enqueues `users.import` in one PostgreSQL
transaction. The worker processes the fixed `users` queue with five workers.
Completed and failed import records are removed after seven days by a daily
02:00 UTC job. Delivery is at least once, so job handlers must remain retry-safe.

Creating a user, synchronously or through an import, also enqueues a
`users.publish-created` job in the same transaction. When
`USER_EVENTS_TOPIC_ARN` is configured, the worker processes the `events` queue
with ten workers and publishes the `user.created` envelope documented in
AsyncAPI. Publication retries retain the same event ID and timestamp; duplicate
SNS delivery remains possible.

Runtime endpoints:

- `GET /livez` checks only that the process can serve HTTP.
- `GET /readyz` performs a bounded PostgreSQL check.
- `GET /openapi.yaml` returns the canonical API contract.
- `GET /asyncapi.yaml` returns the canonical asynchronous message contract.
- `GET /metrics` returns Prometheus metrics; restrict it at the network boundary.

## Commands

```text
make bootstrap          create local config and database
make dev                run the API
make worker             run durable background jobs
make migrate            apply pending migrations
make generate           regenerate OpenAPI and SQL code
make test               run unit tests
make test-integration   run integration tests in an isolated PostgreSQL container
make check              run the full verification suite
make build              build bin/service
make docker-test        smoke-test the production image
make compose-down       stop local PostgreSQL
```

Run `make help` for the authoritative list.

The production binary also exposes a redacted job inspection command. It emits
job metadata but never arguments or user data:

```sh
service jobs list --queue users --state failed --limit 50
```

## Authentication

The API paths in OpenAPI require a bearer token. Set:

```dotenv
APP_ENV=production
AUTH_MODE=oidc
OIDC_ISSUER_URL=https://identity.example.com/
OIDC_AUDIENCE=service-api
```

Issuer matching is exact. The verifier expects a signed JWT accepted by the
OIDC provider's ID-token verifier and requires issuer, audience, expiry,
signature, and a non-empty subject. Confirm that your identity provider issues
compatible JWT access tokens before using this adapter. Authorization policy is
intentionally application-specific and is not part of the template.

`AUTH_MODE=disabled` exists only for development and tests. Startup rejects it
when `APP_ENV=production`.

## Database changes

1. Add a fixed-width `*.up.sql` and matching `*.down.sql` file under
   `db/migrations`.
2. Add feature-owned schema migrations used by SQL generation to `sqlc.yaml`.
3. Add or change named queries in `internal/users/postgres/queries.sql`.
4. Run `make generate` and `make test-integration`.

Migrations run as the explicit `service migrate` command. The API does not
change its database schema during startup. The committed River migrations are
exported from the pinned River v0.35.1 release; upgrade them only through new
application migrations.

## Project boundaries

```text
api/                    canonical HTTP and asynchronous contracts
cmd/service/            thin process entrypoint
db/migrations/          application-wide schema migrations
internal/api/           generated OpenAPI transport contract
internal/app/           runtime composition and shutdown
internal/platform/      shared runtime adapters and infrastructure
internal/users/         users domain behavior and repository contract
internal/users/http/    users HTTP adapter
internal/users/jobs/    users River job adapters
internal/users/postgres/ users SQL, generated access code, and integration tests
```

Generated files are committed. Do not edit `internal/api/api.gen.go` or the
generated Go files under `internal/users/postgres` directly.

The asynchronous contract describes standard SNS delivery: an SQS message body
contains the SNS notification envelope, whose `Message` field contains the
application event. Raw SNS delivery and CloudEvents are intentionally not used
because sibling Node services consume the existing envelope directly.

External publication uses the AWS default credential chain. Configure
`AWS_REGION` and `USER_EVENTS_TOPIC_ARN`; `AWS_ENDPOINT_URL` is available only
for local development and tests and is rejected in production. With no topic in
development, publication jobs remain durably queued while the private `users`
queue continues to run.

## Create a service from the template

After creating a repository from this template, replace the placeholder module
path once:

```sh
make rename MODULE=github.com/acme/my-service
```

Then update `SERVICE_NAME`, the OpenAPI title, and the users example to match the
actual service. Keep package boundaries based on cohesive behavior; do not grow
generic `controllers`, `services`, `repositories`, or `utils` trees.

## Observability and deployment

Logs are text locally and JSON elsewhere. Every response includes
`X-Request-ID`; a valid incoming value is propagated, otherwise one is created.
Set `LOG_LEVEL` to `debug`, `info`, `warn`, or `error`; the default is `info`.
Set `OTEL_EXPORTER_OTLP_ENDPOINT` to enable batched OTLP/HTTP traces. Empty means
no trace exporter. Prometheus HTTP and Go runtime metrics are always available
at `/metrics`, including PostgreSQL readiness availability and check duration.

The production image runs as a non-root distroless user and embeds migrations.
Run the same image with `migrate` before rolling out separate `api` and `worker`
processes; neither runtime role changes the schema at startup. OCI labels expose
the image source, version, revision, and license. `make docker-build` derives
version and revision from the build variables; set `SOURCE_URL` when publishing
from a different repository.
